1.2 Data Types and Variables

This section might feel familiar if you‚Äôve taken a basic programming class (or done your homework, though R and Python differ a bit).

I‚Äôll try to make this brief:

1.2.1 Variable Structures in R

  • Data Frame: A table with rows and columns. Each column can store different data types, making it ideal for organizing datasets.
  • Tibble: A modernized Data Frame. Your data looks prettier when printed and there are no row names by default.
  • Vector: A one-dimensional collection with elements of the same data type.
  • List: An ordered collection that can store mixed data types.
  • Matrix: A two-dimensional Vector with rows and columns, with elements of the same type.
  • Array: Like a Matrix but can handle more than two dimensions.

You‚Äôll most often use Data Frames or Tibbles to create figures, but will use all the variable types at one point or another.

1.2.2 Data Types in R

  • Numeric: The default for numbers, including decimals.
  • Factor: A special structure for categorical data with defined values.
  • Integer: Whole numbers (can be explicitly defined with an ‚ÄúL‚Äù suffix, e.g., 10L).
  • Logical: Boolean values: TRUE or FALSE.
  • Character: Text strings.
  • Complex: Numbers with both real and imaginary components (e.g., 2+3i).
  • Raw: Raw byte data.

Numeric, Factor, and Character variables are essential for plotting data. Factors are especially important ‚Äì we‚Äôll explore them more soon.

These definitions are drawn from this R Workshop GitHub, which inspired a few other sections of this class (thank you, MVesuviusC!).

1.2.3 The str() function

The str() function reveals an object‚Äôs structure.

Let‚Äôs compare a Numeric variable and an Integer by creating two Vectors:

# Create the vectors
numbr <- c(1,2,3)
intg <- c(1L,2L,3L)

# Look at their structure with `str()`
str(numbr)
 num [1:3] 1 2 3
str(intg)
 int [1:3] 1 2 3

Breaking it down, we

  1. Created the variables using <- (the ‚Äúassignment operator‚Äù).
  2. Used the combine function c() to combine numbers/integers into new variables.
  3. Passed each variable to str() to check its structure.

R syntax is typically the function followed by the arguments to the function in parentheses: (arguments).

You can also coerce one data type into another.

# Let's say I actually wanted `numbr` to be an integer:
numbr <- as.integer(numbr)
str(numbr)
 int [1:3] 1 2 3
# Or even a character:
numbr <- as.character(numbr)
str(numbr)
 chr [1:3] "1" "2" "3"

But, oops, just kidding, I actually wanted numbr to be numeric. We can change it back:

numbr <- as.numeric(numbr)
str(numbr)
 num [1:3] 1 2 3
Warning

Be careful not to coerce a variable into an impossible data type!

Note what happens if we try to coerce a character vector into a numeric vector, even if the character could represent a number:

number <- c("one","two","three")
str(number)
 chr [1:3] "one" "two" "three"
number <- as.numeric(number)
str(number)
 num [1:3] NA NA NA

R isn‚Äôt reading and interpreting what our characters might mean, so it isn‚Äôt going to automatically convert one into 1.

If you have numeric variables in your dataset, it‚Äôs best to save them as numbers in your raw data and treat them as either numeric or integer variables in R.

1.2.4 Why use Factors?

Caution

R treats text variables alphabetically by default unless we specify otherwise.

Here‚Äôs an example where that might become confusing.

Say we had a dataset from a contest, where participants could place first, second, third, or fourth.

rankings <- c("first","second","third","fourth")
# Using `str()` shows the expected order...
str(rankings)
 chr [1:4] "first" "second" "third" "fourth"
# But using the `table()` function to summarize the data lists them alphabetically:
table(rankings)
rankings
 first fourth second  third 
     1      1      1      1

How do we fix this to order them logically?

We use the Factor variable type! It lets us specify the expected levels of our categorical data:

rankings <- as.factor(rankings)
# If we just make the variable a factor, we still have the same problem:
str(rankings)
 Factor w/ 4 levels "first","fourth",..: 1 3 4 2
table(rankings)
rankings
 first fourth second  third 
     1      1      1      1 
# Instead, we can explicitly tell it the order we want with the "levels" argument:
rankings <- factor(rankings, levels=c("first","second","third","fourth"))
table(rankings)
rankings
 first second  third fourth 
     1      1      1      1 
# We can set the levels to be whatever we want -- for example, we could reverse them:
rankings <- factor(rankings, levels=c("fourth","third","second","first"))
table(rankings)
rankings
fourth  third second  first 
     1      1      1      1
Extra Credit

Why did str() print the rankings in the correct order the very first time, even though table() showed that R was treating them alphabetically?

Because we entered the names in the logical order when we created the rankings variable.

If we had created the variable with them out of order, str() would print them out of order, but R would still treat them alphabetically:

rankings <- c("third","second","fourth","first")
# Prints the silly order we supplied...
str(rankings)
 chr [1:4] "third" "second" "fourth" "first"
# ...but still treats it alphabetically
table(rankings)
rankings
 first fourth second  third 
     1      1      1      1

Changing the levels of your Factor variables can be very useful when it comes to plotting data, as we will see.

1.3 Getting Help

R has a robust help system ‚Äì just prepend a ? to a function‚Äôs name.

You can search for help in two ways:

  • Single question mark: ?

    • Displays help documentation for a specific function.
    • Example: ?mean() shows the documentation for Base-R‚Äôs mean function.
    • This type of search is best if you know a function‚Äôs name

  • Double question mark: ??

    • Searches help documentation for keyword(s).
    • Example: ??mean() finds all functions (and their documentation) containing mean and returns a list that you can select from.
    • This type of search is useful when you know what you want, but not the exact function name.
    • It‚Äôs also useful to see if there are multiple functions that do similar things (so you can pick the best one for your needs).
    • Works best if you use specific keywords since many functions share common words.

? is especially useful for checking a function‚Äôs arguments (what goes inside the parentheses) and their defaults.

I use ? all the time ‚Äì sometimes just to see what other cool things a function I‚Äôm using can do!

1.4 Packages

Did you notice I specified that mean() is a function in Base-R? R comes ready-made with many essential statistical and graphical functions.

But one of the best things about R is the community of other users who create packages ‚Äì collections of related functions to do specific things in R.

R Packages let you do things like calculate RNA-seq gene expression, plot microbiome phylogenetic trees, use machine learning to create statistical models, and more.

These tasks require diverse skills and expertise to program and execute. Thanks to R‚Äôs broad user base, people make packages to do each of these things (and so much more!).

So how do we get these packages?

  • Almost all packages can be installed with install.packages("package_name").
  • Some bioinformatics tools need to be installed from Bioconductor using BiocManager::install("package_name").
  • Some packages can be installed directly from GitHub using devtools::install_github("package_name").

To load and use a downloaded package, use library(package_name).

Quick notes on packages
  • When installing a package, enclose the name in quotes: ("package_name").
  • When loading a package, you don‚Äôt need the quotes: (package_name).
  • You don‚Äôt need to download a package more than once (though you may need to update it).
  • However, you must load a package every time before you can use it.

If we want to ensure our friend R2 is included, we could recode it so that an NA entry for hair becomes none:

# Recode NA values using replace_na()
# There are many ways to do this, but this is a simple one using the tidyverse package's `mutate` function
small_star_wars <- small_star_wars %>%
  mutate(hair_color = replace_na(hair_color, "none"))

table(small_star_wars$hair_color, small_star_wars$species)

Creating a ggplot is like adding layers to a ‚Äúbase‚Äù graph.

3.1.1 Create the canvas and add aesthetics

3.1.4 Order your variables

You can see R is defaulting to ordering the variables on the x-axis alphabetically.

By chance, they are nearly in perfect descending order, but what if we wanted to order it precisely?

With only six samples, we could coerce name into a factor and then set the levels manually based on the heights we observe. But this is impractical as sample size increases, and also leaves room for human error.

Here‚Äôs a simple code to dynamically reorder a factor based on the value of another variable:

# Use `fct_reorder()` from the `forcats` package to order by `height` (descending)
ggplot(
  small_star_wars,
  aes(
    x=fct_reorder(name, desc(height)),
    y=height,
    fill=species
  )
) +
  geom_col() +
  scale_color_viridis(alpha=0.7, begin = 0, end = 1, option = "D", aesthetics = "fill", discrete = TRUE) +
  theme_light()

4.3 Box plots and Violin plots

When you want to plot a variable‚Äôs distribution, box plots are a good option.

However‚Ä¶

Warning

Box plots can hide important trends in your data.

To illustrate this, I‚Äôm using a dummy dataset (from this excellent explanation of the dangers of box plots).

Let‚Äôs get a better sense of the true distribution by plotting individual points.

To prevent the points from overlapping too much, we can use geom_jitter() to randomly disperse them:

Show the code 
data %>% 
ggplot( aes(x=name, y=value, fill=name)) +
    geom_boxplot() +
    scale_fill_viridis(discrete = TRUE) +
    geom_jitter(color="darkgrey", size=0.7, alpha=0.6) +
    theme_light() +
    theme(
      legend.position="none"
    ) +
    ggtitle("A box plot with jitter") +
    xlab("")

We can see that group C is much smaller, and B seems to have a bimodal distribution, both of which are important to consider when interpreting the data.

A quick warning about ggplot() layers

Because a ggplot is built iteratively, the order you add each geom matters.

Look what happens if we add the jitter first, before the box plot:

data %>%
  ggplot(aes(x = name, y = value, fill = name)) +
  geom_jitter(color = "grey", size = 0.7, alpha = 0.6) + # Jitter first
  geom_boxplot() +                                       # Then box plot
  scale_fill_viridis(discrete = TRUE) +
  theme_light() +
  theme(
    legend.position = "none"
  ) +
  ggtitle("Jitter first, then box plot") +
  xlab("")

The points are plotted first and are covered by the boxes, preventing us from fully visualizing the distribution.

Violin plots provide a fuller view of the distribution:

Show the code 
# sample size
sample_size = data %>% group_by(name) %>% summarize(num=n())

# Plot
data %>%
  left_join(sample_size) %>%
  mutate(myaxis = paste0(name, "\n", "n=", num)) %>%
  ggplot( aes(x=myaxis, y=value, fill=name)) +
    geom_violin(width=1.4) +
    geom_boxplot(width=0.1, color="grey", alpha=0.2) +
    scale_fill_viridis(discrete = TRUE) +
    theme_light() +
    theme(
      legend.position="none",
      plot.title = element_text(size=11)
    ) +
    ggtitle("A violin plot") +
    xlab("")

A downside of violin plots is that they don‚Äôt convey differences in sample size very well.

5.4 Heatmaps

We‚Äôll close by making a heatmap.

This is one of the few instances where ggplot2 isn‚Äôt your best plotting option.

While you can use it to make heatmaps, the appropriately named pheatmap (for ‚ÄúPretty Heatmap‚Äù) is best.

Here‚Äôs a simple heatmap comparing height and weight in our dataset:

This looks neat, but let‚Äôs make some tweaks to make it more interpretable.

5.4.1 Scaling

First, height (cm) and weight (kg) are in different units, but the color scale is treating them the same.

To address this, we can scale by column, centering and scaling each variable so that they have a mean of zero and standard deviation of one.

Show the code 
pheatmap(
  heatmap_data,
  cluster_rows = FALSE,   # Do not cluster rows (optional)
  cluster_cols = FALSE,   # Do not cluster columns (optional)
  scale = "column",
  main = "Heatmap of Height and Weight"
)

The interpretation of the data doesn‚Äôt change (the high values are still high, and the low values are still low), but now all the heights don‚Äôt appear higher than all the weights.

Caution when scaling

Be careful that you are scaling the appropriate values if you apply a scaling factor.

Look what happens if we scale by row instead of by column:

pheatmap(
  heatmap_data,
  cluster_rows = FALSE,
  cluster_cols = FALSE,
  scale = "row",
  main = "Heatmap of Height and Weight"
  
)

Because the mean of height is greater than the mean of weight (because they are on different scales), all the heights are greater than all the weights.

Not very informative!

5.4.2 Clustering

Let‚Äôs also apply a clustering algorithm to determine which individuals are more similar, based on these two variables:

Show the code

  cluster_rows = TRUE,   
  cluster_cols = FALSE,   
  scale = "column",
  main = "Heatmap of Height and Weight"
)

By looking at the dendogram on the left, we can get a greater appreciation for which individuals it is grouping together.

For example, Padm√© and Leia are grouped together, as are R2-D2 and R5-D4.

Hmm, that‚Äôs interesting. Padm√© and Leia are mother and daughter, and R2-D2 and R5-D4 are both the same kind of (astromech) droid.

So does that mean all the groupings are meaningful?

Well, we can also see that Darth Vader and IG-88 are grouped together. While Darth Vader is ‚Äúmore machine than man‚Äù according to Obi-Wan, he‚Äôs still a human, but is being grouped together with an assassin droid.

Plus, there‚Äôs a whole mess of other groupings in the middle that are difficult to assess by eye.

5.4.3 Annotating

This is where annotations come in handy.

Let‚Äôs create an annotation for species:

Show the code 
# Create a species annotation dataframe
species_annotation <- data.frame(Species = heatmap_starwars$species)
rownames(species_annotation) <- heatmap_starwars$name  # Match the rownames of the heatmap data

# Define colors for the species annotation
annotation_colors <- list(
  Species = c(
    "Human" = "#1f77b4",  # Blue
    "Droid" = "#ff7f0e",  # Orange
    "Gungan" = "#2ca02c" # Green
  )
)

# Generate the heatmap with annotation
pheatmap(
  heatmap_data,
  annotation_row = species_annotation,  # Add row annotations
  annotation_colors = annotation_colors,  # Add colors for the annotations
  cluster_rows = TRUE,  # Cluster rows
  cluster_cols = FALSE,
  scale="column"
)
